# SlaySP2Manager 产品需求文档

## 1. 产品定位

SlaySP2Manager 是一个基于 Rust + Tauri 的 Slay the Spire 2 桌面端模组管理器。

它不应该只是一个“把文件复制进 `mods` 文件夹”的工具，而应该成为玩家进入 STS2 Mod 世界的默认入口。

这个产品要解决的不是“能不能装上”，而是以下几个高焦虑场景：

- 第一次装 Mod，不知道游戏目录和存档目录在哪
- 想找新 Mod，但不想在浏览器和本地文件夹之间来回切
- 安装或更新后，不确定有没有冲突
- 原版存档和模组存档容易混淆
- 想切换不同玩法方案，但怕弄乱现有环境
- 更新失败后，缺少回滚手段

理想状态是：

- 新手 2 分钟内完成首次安装
- 老玩家能稳定管理多套 Mod 组合
- 整个界面安静、克制、明确，不像脚本壳，也不像杂乱后台

## 2. 产品原则

这个产品不只是把功能做全，而是要把用户最容易出错的关键瞬间做得极其顺滑。

设计原则：

- 先降低焦虑，再增加功能
- 每个页面只保留一个主动作
- 默认界面只展示用户此刻必须知道的信息
- 把复杂度折叠到第二层，而不是堆在首页
- 文件安装、更新、存档迁移都按“事务”处理，而不是直接写盘
- 所有高风险动作都必须有预览、确认、备份、回滚

具体设计约束：

- 不做仪表盘式信息拼盘
- 不做重阴影和强游戏风格视觉
- 不做大量无意义状态提示
- 不让用户先看说明文档再会用产品

## 3. 目标用户

### 3.1 核心用户

- 第一次安装 STS2 Mod 的玩家
- 只装少量 QoL Mod 的玩家
- 在原版与模组模式之间切换的玩家

### 3.2 进阶用户

- 维护多套 Mod 方案的重度玩家
- 想导出和分享整套环境的玩家
- 需要稳定联机或直播环境的玩家

## 4. 核心问题

- 用户不知道游戏路径和存档路径
- 当前 Mod 管理方式依赖脚本、压缩包和手动操作
- Mod 的安装、启用、禁用、卸载没有统一心智模型
- 远程发现 Mod 的体验分散在 Nexus 页面和本地目录之间
- 更新后容易破坏稳定环境
- 原版存档和模组存档的边界不够清晰

## 5. 产品目标

### 5.1 用户目标

- 在应用内完成找 Mod、装 Mod、更新 Mod、切换方案的完整闭环
- 随时知道当前装了什么、启用了什么、哪里有风险
- 在原版和模组玩法之间切换时，不丢进度、不搞混存档
- 出问题时可以快速恢复

### 5.2 产品目标

- 成为 STS2 玩家默认使用的桌面端模组入口
- 把常见支持问题前置成产品能力，而不是靠说明文档
- 为后续的 profile 分享、推荐、兼容性系统打基础

## 6. V1 不做什么

- 不做通用的模组配置编辑器
- 不做完全自动的兼容性修复
- 不做社区评论、评分、社交系统
- 不做过度复杂的 load order 系统，除非 STS2 生态明确需要

## 7. 版本范围

### 7.1 V1 MVP

- 自动检测游戏目录
- 自动检测存档目录和 Steam 用户目录
- 扫描本地已安装 Mod
- 安装本地 zip
- 启用和禁用 Mod
- 卸载 Mod
- 冲突检测和版本识别
- 原版与模组存档双向复制
- 自动备份与恢复
- 连接 Nexus API Key
- 在应用内搜索 Nexus 上的 STS2 Mod
- 查看 Mod 详情
- 检查已安装 Mod 的远程更新
- 一键更新
- 支持 2 到 3 套本地 Profile

### 7.2 V1.5

- 批量更新
- Profile 导入导出
- 下载历史
- 推荐 Mod
- 更完善的诊断页
- 回滚记录

### 7.3 V2

- 依赖图谱
- 兼容性规则系统
- 整套收藏或合集能力
- 社区共享 Profile
- 更细粒度的组合校验

## 8. 信息架构

一级导航严格控制为 5 个：

1. 模组库
2. 发现
3. 配置方案
4. 存档
5. 设置

这样既能覆盖完整能力，又不会让产品看起来像后台系统。

## 10. 核心页面设计

## 10.1 模组库首页

首页不是数据大屏，而是状态控制中心。

推荐布局：

- 顶部状态条：游戏状态、当前 Profile、更新状态
- 中央主区域：已安装 Mod 列表
- 右侧窄栏：需要处理
- 底部：最近操作

右侧窄栏只放 4 类信息：

- 可更新
- 冲突
- 缺失文件
- 存档风险

用户打开应用后，第一眼就知道应该先处理什么。

## 10.2 发现页

推荐布局：

- 顶部是搜索框
- 下方是筛选条件
- 左侧是搜索结果列表
- 右侧是 Mod 预览面板

支持的筛选维度：

- 分类
- 最近更新
- 热门
- 已安装
- 未安装
- 作者

主动作：

- 安装

次级动作：

- 在 Nexus 中查看
- 稍后再看
- 添加到当前方案

## 10.3 Mod 详情页

页面结构：

- 顶部：名称、作者、版本、更新时间
- 摘要
- 文件版本
- 兼容性提示
- 本地状态
- 操作区

操作根据状态变化：

- 安装
- 更新
- 禁用
- 启用
- 卸载
- 打开 Nexus 页面

## 10.4 配置方案详情页

页面结构：

- 方案名称和说明
- 包含的 Mod
- 启用数量
- 最近应用时间
- 已知风险

操作：

- 应用方案
- 重命名
- 复制
- 导出
- 删除

## 10.5 存档页

建议采用双栏布局：

- 左边显示原版存档槽位
- 右边显示模组存档槽位

中间只放关键动作：

- 复制到模组存档
- 复制到原版存档
- 备份
- 恢复

任何高风险动作都必须显示预览：

- 来源槽位
- 目标槽位
- 是否会覆盖
- 备份将保存到哪里

## 11. 关键交互流程

## 11.1 首次启动流程

1. 启动应用
2. 自动检测游戏目录
3. 如果没找到，引导用户选择包含 `SlayTheSpire2.exe` 的目录
4. 自动检测存档目录
5. 扫描本地已安装 Mod
6. 进入一个极简的准备完成页面
7. 页面只给出三个主动作：
8. 连接 Nexus
9. 导入本地 zip
10. 浏览 Mod

成功标准：

- 用户在不看教程的情况下完成首次准备

## 11.2 从发现页安装 Mod

1. 打开发现页
2. 搜索关键词或浏览推荐
3. 进入 Mod 详情页
4. 点击安装
5. 弹出安装预览
6. 展示安装目标、冲突、现有版本、回滚点
7. 用户确认
8. 安装完成后刷新模组库
9. 可选提示：是否添加到当前方案

## 11.3 导入本地 zip

1. 用户拖拽 zip 到窗口，或手动选择文件
2. 解析压缩包
3. 读取 manifest、ID、名称、版本
4. 与本地已安装 Mod 进行比较
5. 如果冲突，展示冲突处理选项
6. 执行安装
7. 根据安装模式，默认进入启用或禁用状态

## 11.4 更新已安装 Mod

1. 模组库展示可更新状态
2. 用户进入可更新列表
3. 应用比较本地和 Nexus 的版本
4. 用户点击更新
5. 创建恢复点
6. 下载、校验、应用更新
7. 将结果写入活动记录
8. 如失败，自动回滚

## 11.5 从原版切换到模组玩法

1. 用户应用一套模组方案
2. 如果模组存档为空或明显落后，应用弹出提醒
3. 建议把原版存档复制到模组存档
4. 在复制前自动备份
5. 应用完成后，明确提示当前方案和当前存档状态

## 11.6 回到原版玩法

1. 用户应用原版方案，或禁用全部 Mod
2. 应用提醒：当前进度可能还在模组存档中
3. 提供“模组存档复制回原版”的引导
4. 先备份后覆盖
5. 回到稳定的原版状态

## 12. Nexus 集成策略

发现和更新功能应以 Nexus 官方 API 为基础，而不是以抓取网页为基础。

V1 中应支持的能力：

- 按关键词搜索 STS2 Mod
- 查看 Mod 详情
- 比较本地与远程版本
- 使用用户自己的 API Key 下载与更新

建议的发现模块：

- 搜索
- 相似 Mod
- 最近更新
- 热门或新发布

后续推荐逻辑可以基于：

- 已安装 Mod 的标签
- 作者重合
- 当前方案类型

## 14. 状态模型

每个 Mod 都应该有明确状态标签：

- 已安装
- 已启用
- 已禁用
- 可更新
- 有冲突
- 已损坏
- 未知

每个高风险操作都应有明确预检状态：

- 安全
- 需要备份
- 检测到冲突
- 文件占用中
- 需要二次确认

## 15. 视觉方向

界面应更像一个高级桌面工具，而不是游戏启动器。

建议视觉语言：

- 偏暖的浅色背景，不用纯白
- 石墨黑或墨色文本
- 只保留一种克制的强调色，建议深蓝或暗金二选一
- 以细边框代替重阴影
- 使用明确的大间距节奏
- 保留轻量且短促的过渡动画

明确避免：

- 霓虹游戏风
- 大量卡片堆叠
- 强渐变背景
- 厚重系统边框
- 过量图标

## 16. 文案原则

文案必须：

- 短
- 准
- 平静
- 面向动作

推荐写法：

- `安装更新`
- `复制原版存档到模组存档`
- `已创建备份`
- `发现冲突，需要选择`

避免：

- 很长的警告段落
- 论坛式解释
- 情绪化系统提示

## 17. 成功指标

### 17.1 产品指标

- 首次启动到首次成功安装的耗时
- 安装成功率
- 更新成功率
- 回滚成功率
- 因存档混淆产生的问题数

### 17.2 体验指标

- 首次安装需要的步骤数
- 每次成功任务所触发的警告次数
- 用户在安装流程中离开应用去浏览器的次数

## 18. 待决策问题

- 远程安装后的默认状态是启用还是禁用
- 是否要维护本地下载缓存
- Profile 存精确版本还是浮动版本
- Nexus 元数据要缓存多少到本地
- 是否允许用户在未配置 API Key 的情况下先浏览部分内容

## 19. V1 推荐决策

- 如果冲突风险较高，远程安装默认先禁用
- 维护受控的本地下载缓存，并提供清理规则
- Profile 尽量记录精确版本
- 本地只持久化必要元数据，用于状态、速度和离线展示
- 如果 API 允许，可提供有限浏览；下载和完整同步则要求用户配置 API Key
